# winnow

An opinionated observability UI built on Quickwit. Born from frustration with Grafana.

## Project Context
- See `docs/CONTEXT.md` for the Grafana pain points that motivated this
- See `docs/ARCHITECTURE.md` for system design
- See `docs/ROADMAP.md` for what we're building

## Stack
- **Backend:** Zig
- **Frontend:** React + TypeScript + shadcn/ui
- **Storage:** Quickwit (traces, logs, AND metrics — one backend for everything)
- **Build/Dev:** Nix flake

## Rules
- **Never run git write operations.** No `git add`, `git commit`, `git push`, `git checkout`, `git reset`, etc. Read-only commands (`git status`, `git diff`, `git log`) are fine.
- **Always update `docs/TODO.md`** after completing a phase or significant chunk of work. Check off finished items, note any remaining verification steps.

## Conventions
- Nix for all build/dev tooling, no global installs
- Single binary deployment (backend serves frontend assets)
- No over-engineering — build what we need now, not what we might need later
- Minimal dependencies — a small set of mature, explicit deps (zig-protobuf, gRPC-zig), not half the universe

## Key Design Decisions
- One unified interface, not dashboards + explore + drilldown
- No raw query editors in the UI
- Every data source is fully supported or not supported at all
- Service map is the primary navigation entry point
- Everything is connected: service → traces → logs → spans, all one click

## Development

### Prerequisites
Everything is managed by Nix. Enter the dev shell:
```
nix develop
```

### Building
```
# Full build (includes proto gen + tests + binary):
nix build

# Dev iteration (inside nix develop):
cd backend
zig build gen-proto   # generate Zig from .proto files (needs protoc from devShell)
zig build             # compile the server
zig build run         # compile and run
zig build test        # run tests
```

### Important: always use the Nix dev shell
All build commands (`zig build`, `protoc`, etc.) require tools from the Nix devShell. Either:
- Run inside `nix develop`, or
- Prefix with `nix develop --command bash -c "..."`

`nix build` handles this automatically — it runs in an isolated sandbox with all deps.

### Protobuf workflow
OTLP `.proto` files are vendored in `backend/proto/`. Generated Zig code goes to `backend/src/proto/` (gitignored). To regenerate:
```
cd backend && zig build gen-proto
```

The gen-proto step builds `protoc-gen-zig` from the zig-protobuf dependency and uses system `protoc`. We build protoc-gen-zig ourselves rather than using zig-protobuf's `RunProtocStep.create()` because it resolves internal paths relative to the consumer project.

### Zig dependency management
Zig deps are declared in `backend/build.zig.zon`. For nix builds, deps are locked in `backend/build.zig.zon2json-lock` (generated by zig2nix). If you add or update a zig dependency:
```
# 1. Add/update the dep in build.zig.zon
# 2. Regenerate the nix lock file:
nix run github:Cloudef/zig2nix -- zon2lock backend/build.zig.zon
# 3. Stage the lock file (nix ignores unstaged files in git repos)
git add backend/build.zig.zon2json-lock
```

### Shell gotcha
`cd` is aliased to zoxide in this environment, which fails in non-interactive shell invocations. Wrap in `bash -c '...'` when scripting.
